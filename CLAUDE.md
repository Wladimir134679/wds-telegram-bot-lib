# CLAUDE.md — инструкции для AI-агента по проекту `wds-telegram-bot-lib`

Этот файл описывает, **что агенту можно делать, а что нельзя**, и даёт контекст о
проекте. Читай его перед началом любой работы. Всё общение, комментарии и
Java-документация — **на русском языке**.

---

## 0. Главное правило: агент НИЧЕГО не запускает

Сборку, компиляцию и запуск проекта выполняет **только пользователь-разработчик
руками**. Агент пишет и правит код, но не исполняет его.

**Агенту ЗАПРЕЩЕНО запускать:**

- сборку и задачи сборки (`gradle`, `gradlew`, `./gradlew build`, `assemble`,
  `jar`, `bootJar` и любые другие Gradle-таски);
- компиляцию (`compileJava`, `javac` и т.п.);
- **тесты** — это особенно важно: никаких `test`, `gradlew test`, запусков
  JUnit, отдельных тестовых классов и т.д.;
- запуск приложения (`bootRun`, `java -jar`, `SpringApplication.run` и пр.);
- любые команды, которые скачивают зависимости, прогревают кэш Gradle или
  «проверяют, что собирается».

**Почему так:** проверку сборки/компиляции/тестов делает разработчик у себя.
Агент не должен тратить время и ресурсы на запуск и не должен полагаться на
«зелёный прогон» как на подтверждение. Если хочется проверить корректность —
анализируй код глазами, рассуждай о типах и контрактах, но **не исполняй**.

**Что агенту делать вместо запуска:**

- если нужно убедиться в работоспособности — словами опиши, какие команды
  пользователь может выполнить сам, и что он должен увидеть;
- по умолчанию НЕ предлагай команды запуска без явной просьбы;
- завершай задачу формулировкой вида «код готов, собери и протестируй на своей
  стороне», а не «я собрал и проверил».

> Исключение — только прямая и явная просьба пользователя выполнить конкретную
> команду. Тогда выполняй ровно её и ровно один раз, без «заодно соберу/протестирую».

## 1. Ввод пользователя — голосовой. Транскрибация бывает неточной

Пользователь надиктовывает задания голосом, и распознавание речи **иногда
ошибается**: путает похожие по звучанию слова, теряет окончания, вставляет
посторонние или бессмысленные в контексте термины.

**Как с этим работать:**

- Любое **странное, не подходящее по смыслу слово ставь под сомнение** и
  реконструируй, что пользователь скорее всего хотел сказать, исходя из
  контекста проекта (Spring, Telegram-бот, команды, сессии, обработчики).
- Особенно внимательно относись к:
  - англоязычным техническим терминам, записанным на слух
    (например, услышанное «эм-си-пи / MCP» в контексте веб-обработчиков скорее
    всего означает модель «как **Spring MVC**», а не что-то иное);
  - аббревиатурам и именам классов/аннотаций;
  - числам, единицам измерения (секунды/минуты), названиям версий.
- Если есть **очевидная и уверенная** интерпретация — применяй её, но
  **коротко отметь в ответе**, как ты понял спорное место («"года сборки" понял
  как "задачи сборки"»).
- Если двусмысленность **существенная** и влияет на результат — переспроси
  одним точным вопросом, а не делай наугад.
- Не воспринимай дословно фрагменты, которые грамматически «рассыпаны»:
  восстанавливай намерение, а не букву.

## 2. Что это за проект

`wds-telegram-bot-lib` — **библиотека (Spring Boot starter)** для написания
Telegram-ботов. Подключаешь зависимость — и пишешь «контроллеры команд» почти
как в Spring MVC: с аннотациями и инъекцией параметров по типу. Транспорт —
`telegrambots-longpolling` 10.0.0 (long polling, без webhook).

Это именно **библиотека**, а не приложение:

- у неё **не должно быть** боевого `main()`/`@SpringBootApplication` как точки
  входа для потребителя (текущий `WdsTelegramBotLibApplication` — временный
  каркас от генератора, целевая модель — `@AutoConfiguration` + `@ConditionalOn*`);
- бины подключаются через авто-конфигурацию и условия, а не через навязанный
  потребителю `@ComponentScan`.

**Стек:** Java 21, Spring Boot 4.1.0, Gradle (Kotlin DSL), Lombok,
`telegrambots-longpolling` / `telegrambots-client` 10.0.0.

## 3. Ключевые особенности архитектуры (ядро подхода)

Это то, ради чего проект существует. Сохраняй и развивай именно эти концепции.

### 3.1. Модель «как Spring Web MVC» — контроллеры команд

Потребитель библиотеки описывает обработчики команд декларативно (аннотации на
бинах и методах), а библиотека **маршрутизирует** входящие `Update` в нужный
метод и **инъектирует аргументы по типу** (`Update`, `Message`, `CallbackQuery`,
`Chat`, сессия пользователя и т.д.). Цель — читаемые сигнатуры методов-команд,
минимум шаблонного кода, ощущение «контроллеров Spring MVC», но для Telegram.

### 3.2. Цепочка обработчиков с приоритетами (промежуточная обработка)

Каждый входящий `Update` прогоняется через **список обработчиков, отсортированных
по приоритету**. Это позволяет вставлять промежуточную обработку, не трогая
существующий код: логирование, метрики, антифлуд, предобработку, и наконец —
маршрутизацию команд. Новый этап обработки = новый обработчик в цепочке с нужным
приоритетом.

Требования к реализации цепочки:

- порядок задаётся приоритетом (рассмотреть Spring `Ordered`/`@Order`);
- **изоляция ошибок**: исключение в одном обработчике не должно ронять обработку
  всего апдейта (оборачивать каждый обработчик в try/catch);
- не мутировать инжектированный Spring'ом список (копировать перед сортировкой).

### 3.3. Сессии по чату/пользователю с TTL

Для каждого чата (и пользователя внутри чата) хранится **сессия о последних
нескольких шагах диалога**. Сессия **живёт ограниченное время** (TTL) и
вычищается фоново по истечении. Это даёт командам контекст предыдущих действий
пользователя без внешнего хранилища.

Требования к реализации сессий:

- единицы измерения TTL должны быть **согласованы по всей цепочке** (конфиг →
  логи → планировщик). Предпочтительно `Duration`, а не «голый» `int` без единиц;
- потокобезопасность — единообразная: `computeIfAbsent` вместо `get`-затем-`put`,
  без бессмысленного `synchronized` поверх `ConcurrentHashMap`;
- интервалы фоновой очистки не должны вырождаться в `0` (никаких `ttl/4`, дающих
  ноль при малых значениях → `IllegalArgumentException`);
- сессии — **опциональная** фича (включается флагом), с no-op реализацией по
  умолчанию, чтобы убрать россыпь null-проверок.

### 3.4. Callback-данные

Telegram ограничивает `callback_data` **64 байтами**. Не сериализуй в него
«толстый» JSON. Используй компактный формат (`cmd:arg`), короткие коды или
храни полезную нагрузку на сервере, а в callback клади короткий ключ.
**Всегда валидируй длину ≤ 64 байта** при формировании кнопки.

## 4. Контекст: перенос из старого проекта

Рядом лежат файлы разбора старого проекта-прототипа — читай их как источник
требований и предостережений (что хорошо, что плохо, что чинить при переносе):

- `REVIEW_00_OVERVIEW.md` — обзор и архитектура, главные проблемы;
- `REVIEW_01_CLASSES.md` — поклассовый разбор;
- `REVIEW_02_MIGRATION.md` — план переноса и исправлений;
- `REVIEW_03_SECURITY.md` — вопросы безопасности.

Новый код должен **повторять удачные идеи** старого (starter, контроллеры команд,
цепочка обработчиков, сессии) и **не повторять его ошибки** (старый стиль
авто-конфигурации, рефлексия без кэша, баги потокобезопасности и единиц времени,
логирование секретов и PII, JSON в `callback_data`).

## 5. Стиль кода и документация

- **Язык:** комментарии и Javadoc — **на русском**. Технические идентификаторы
  (имена классов, методов, аннотаций) — в оригинале.
- **Javadoc обязателен** для публичного API библиотеки: интерфейсы обработчиков,
  аннотации команд, точки расширения, конфигурационные свойства. Описывай
  назначение, контракт, потокобезопасность и поведение по умолчанию.
- **Безопасность логов:** никогда не логируй токен бота, целиком `Update`, тексты
  сообщений пользователей на уровне INFO. Секреты маскируй. Подробности — DEBUG.
- **Lombok:** один согласованный стиль, без дублирующих аннотаций
  (`@Data` уже включает getter/setter/toString — не навешивай их сверху).
- **Авто-конфигурация:** `@AutoConfiguration` + `@ConditionalOnMissingBean` /
  `@ConditionalOnProperty`, регистрация через `META-INF/spring/...imports`.
  Не использовать `@ComponentScan`/`@EntityScan` в стартере.
- **Сборка библиотеки:** это библиотека, поэтому fat/исполняемый jar не нужен —
  при правках сборки помни про разницу между `jar` и `bootJar` (но саму сборку
  **не запускай** — см. раздел 0).
