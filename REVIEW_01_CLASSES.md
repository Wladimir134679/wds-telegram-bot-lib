# Разбор старого проекта — поклассово

Формат: для каждого класса — **назначение**, **что хорошо**, **что плохо**,
**что делать при переносе**. Файлы сгруппированы по слоям.

---

## 1. Загрузка / авто-конфигурация

### `WdsTelegramBotApplicaton`
**Назначение:** точка входа Spring Boot (`main`).
**Плохо:**
- Опечатка в имени (`Applicaton`).
- В **библиотеке** не должно быть `main()`, `@SpringBootApplication`,
  `@EnableConfigurationProperties` на уровне приложения.
- `log.error("not executing lib")` — заглушка-костыль.
**Перенос:** удалить. У библиотеки нет точки входа. Если нужен демо-/тестовый
запуск — вынести в отдельный модуль `example`/`sample`, не в библиотеку.

### `InitTelegramBotStarter`
**Назначение:** конфигурация стартера, регистрирует бины через `@ComponentScan`.
**Плохо:**
- `@ComponentScan("ru.wdeath...")` в стартере — антипаттерн: навязывает
  сканирование своего пакета и побочно может цеплять чужое.
- `@EntityScan` бессмысленен (JPA нет).
- В `AutoConfiguration.imports` указан этот класс, но он помечен
  `@Configuration`, а не `@AutoConfiguration` → это не настоящая авто-конфигурация.
- `@Slf4j` не используется.
**Перенос:** заменить на `@AutoConfiguration` класс, который **явно** объявляет
бины через `@Bean` с `@ConditionalOnMissingBean`/`@ConditionalOnProperty`.
Регистрировать команды/обработчики не сканированием, а через интерфейсы,
реализованные в коде потребителя (Spring сам соберёт их в `List<...>`).

### `META-INF/spring/...AutoConfiguration.imports`
**Хорошо:** правильный механизм для Spring Boot 3+/4 (вместо старого
`spring.factories`).
**Плохо:** указывает на `@Configuration`, а не `@AutoConfiguration`.
**Перенос:** оставить файл, но привести содержимое к настоящему
`@AutoConfiguration`-классу.

## 2. Конфигурация / свойства

### `TelegramBotProperty`
**Назначение:** `@ConfigurationProperties("telegram.bot")` — токен, username,
настройки сессий.
**Хорошо:** типобезопасные свойства — правильный путь.
**Плохо:**
- `@PostConstruct` логирует `this` через `@ToString` → **в лог попадает токен**.
- `@Getter @Setter @ToString` на классе + `@Data` на вложенных `TimeLife`/`Session`
  — дублирование и несогласованность (`@Data` уже включает гso/setter/toString).
- `session` может быть `null` → NPE в `SessionManagerServiceImpl.init()`.
- Поля `int` для TTL без указания единиц измерения (мин? сек?) — отсюда баг.
**Перенос:** убрать логирование секретов (или маскировать токен). Использовать
`Duration` вместо `int` для TTL. Дать дефолты для `session`. Один стиль Lombok.
Рассмотреть Java `record` + `@ConfigurationProperties` (конструкторная привязка).

### `TelegramBotConfig`
**Назначение:** отдельный YAML-файл конфигурации, читается вручную SnakeYAML.
**Плохо:**
- Изобретает велосипед поверх `@ConfigurationProperties`.
- `System.out.println` вместо логгера.
- `static final Yaml YAML` — SnakeYAML `Yaml` **не потокобезопасен**, а тут общий.
- Создаёт пустой файл и тут же читает его → `valueMap == null` для пустого файла,
  `get()` молча возвращает `null`.
- «Config», но писать/сохранять не умеет. `get` возвращает `Object`.
**Перенос:** скорее всего выбросить. Если нужна runtime-перезагрузка
пользовательских настроек — сделать осознанный компонент с логгером,
потокобезопасностью и типобезопасным доступом, либо опереться на Spring
`@RefreshScope`/Environment.

## 3. Движок и жизненный цикл

### `TelegramLongPollingEngine`
**Назначение:** реализует `LongPollingSingleThreadUpdateConsumer`, оборачивает
`TelegramClient`, рассылает апдейты обработчикам.
**Хорошо:** центральная точка; сортировка обработчиков по приоритету; удобные
`execute`.
**Плохо:**
- Создаёт `OkHttpTelegramClient` прямо в конструкторе → невозможно
  замокать/подменить, сложно тестировать.
- **Мутирует инжектированный `List<HandlerBotUpdate>`** (`handlerBotUpdates.sort(...)`
  на месте) — побочный эффект над spring-managed коллекцией.
- `executeNotException` назван «без исключения», но ловит и **бросает**
  `RuntimeException`; переменная `Serializable execute` не используется.
- `consume` логирует весь `Update` на INFO — шум и PII.
- `consume` использует `forEach` без try/catch → одно исключение рушит всю цепочку.
- Смешение ролей: и «обёртка клиента», и «consumer», и держатель `botSession`
  (мутабельное общее состояние через `@Setter`).
**Перенос:** разделить ответственности — отдельный «клиент/sender», отдельный
«диспетчер обработчиков». Клиент инжектировать как бин (фабрика по токену).
Не мутировать инжектированный список (копировать). Обернуть каждый обработчик в
try/catch. Понизить уровень логов, не логировать целиком `Update`. Переименовать
`executeNotException` (например, `executeQuietly` и реально глотать/возвращать
`Optional`/`boolean`).

### `HandlerBotUpdate`
**Назначение:** интерфейс обработчика апдейта + `priority()`.
**Хорошо:** простое расширение цепочки.
**Плохо:** `priority()` своим `int` дублирует Spring'овый `Ordered`/`@Order`.
Передача `engine` в каждый метод — лишняя связанность.
**Перенос:** рассмотреть `extends Ordered` или `@Order`. Подумать, не заменить ли
`engine` на инжектируемый sender, чтобы обработчик не зависел от движка.

### `HandlerUpdatePriority`
**Назначение:** константы приоритетов.
**Плохо:** нет `private` конструктора/`final` класса; «магические» числа.
**Перенос:** мелочь; либо `final` + приватный конструктор, либо `enum` с
порядком, либо `@Order` на бинах.

### `SystemHandlerTelegramServiceImpl`
**Назначение:** обработчик-логгер (пример).
**Плохо:** имя `...ServiceImpl` вводит в заблуждение — это обработчик, не сервис.
Логирует на INFO на каждый апдейт.
**Перенос:** переименовать (`LoggingUpdateHandler`), уровень DEBUG, сделать
опциональным через property.

### `ProcessingCommandBotService`
**Назначение:** обработчик, делегирующий в `ManagerCommandAndSessionGroupService`.
**Хорошо:** тонкий, по делу.
**Плохо:** опять `...Service` для обработчика.
**Перенос:** оставить как есть, переименовать для единообразия.

### `TelegramBotApiStarterService`
**Назначение:** регистрация и остановка бота.
**Плохо:**
- `@PostConstruct postConstructor()` **и** `@EventListener(...Start)` зовут один
  `startBot()` → потенциальный двойной старт.
- Старт в `@PostConstruct` — контекст может быть ещё не готов.
- Ошибка регистрации только логируется (`botSession` останется `null`), потом
  `stopBot()` → **NPE** на `getBotSession().stop()`.
- `new TelegramBotsLongPollingApplication()` создаётся в конструкторе (не бин,
  не закрывается явно).
**Перенос:** перейти на `SmartLifecycle` (start/stop в рамках жизненного цикла
контекста). Убрать дублирование старта. Корректно обрабатывать неуспешную
регистрацию и null-сессию. `TelegramBotsLongPollingApplication` сделать бином и
закрывать в `stop()`.

### `events/TelegramBotEventStart`, `TelegramBotEventStop`
**Назначение:** маркер-события start/stop.
**Плохо:** `@Data @ToString` на **пустых** классах бессмысленны; не наследуют
`ApplicationEvent` (хотя для современного Spring это и не обязательно).
**Перенос:** заменить на `record`-события или оставить пустыми POJO без лишних
аннотаций. Задокументировать, кто и когда их публикует.

## 4. Маршрутизация команд

### Аннотации `@CommandNames`, `@CommandFirst`, `@CommandOther`, `@ParamName`
**Хорошо:** аккуратный декларативный DSL; `@CommandNames` с `value()` + `type()`
(MESSAGE/CALLBACK) — удобно.
**Плохо:**
- `@ParamName` требует имена параметров в рантайме → нужен флаг компиляции
  `-parameters`, иначе `param.getName()` вернёт `arg0`.
- `@CommandFirst`/`@CommandOther` без параметров — вся маршрутизация «первая/
  последующая» держится на наличии одного метода; неочевидно при нескольких методах.
**Перенос:** сохранить идею. Документировать необходимость `-parameters` или
перейти на явные имена в аннотации. Продумать поведение при нескольких
аннотированных методах (запрещать/брать первый/по порядку).

### `TypeCommand` (enum MESSAGE/CALLBACK/NONE)
**Хорошо:** простой и понятный.
**Перенос:** оставить.

### `CommandContext`
**Назначение:** DTO контекста вызова команды.
**Плохо:** `@Data` мутабельный; `data` имеет тип `Object` (нетипизировано).
**Перенос:** сделать по возможности иммутабельным (билдер/record), типизировать
полезную нагрузку или хотя бы документировать формат.

### `CommandContainer`
**Назначение:** хранит контроллер команды и вызывает его методы рефлексией с
инъекцией аргументов.
**Плохо:**
- `@Data` на классе с массивами и `Method` → странные `equals/hashCode`.
- `executeFirst/executeOther` на **каждый вызов** делают
  `getClass().getDeclaredMethods()` и линейный поиск — нет кэша.
- Порядок `getDeclaredMethods()` **не гарантирован**; при нескольких
  `@CommandFirst` вызовутся все.
- Поля `commandFirst, commandOther` объявлены, но **не используются** (мёртвый код).
- Инъекция аргументов хардкодит ограниченный набор типов; кастомные параметры —
  только `Long` (`userId/chatId/messageId`), нет `String`-аргументов команды,
  `int`, и т.п.
- `getObjectLongParameter` опирается на имена параметров (хрупко).
- Исключения `invoke` оборачиваются в `RuntimeException`, теряя контекст.
**Перенос:** при регистрации команды **один раз** найти и закэшировать нужные
`Method` (+ их преднасчитанные резолверы параметров). Запретить/детерминировать
несколько аннотированных методов. Расширить набор резолверов параметров
(текстовые аргументы команды, числа, кастомные конвертеры). Аккуратно
прокидывать исключения пользовательского кода.

### `ManagerCommandContainerService`
**Назначение:** на старте находит бины с `@CommandNames` и строит карту
команда→контейнер.
**Хорошо:** один проход при старте через `getBeansWithAnnotation`.
**Плохо:**
- Логика слияния контейнеров по совпадающему имени (`loadTextCommand`) хрупкая;
  при пересечении имён у разных классов поведение неочевидно (перезапишется
  `objectController`).
- `@Autowired(required = false)` для сессий — ок, но дальше всюду null-проверки.
- `getDeclaredMethods` фактически дёргается уже в `CommandContainer` на каждый вызов.
**Перенос:** сделать регистрацию явной и детерминированной; на этапе старта
сразу резолвить методы. Падать с понятной ошибкой при конфликте имён.

### `ManagerCommandAndSessionGroupService`
**Назначение:** строит `CommandContext` из `Update`, связывает с сессиями,
выбирает «первую»/«последующую» команду.
**Плохо:**
- Длинные методы, переплетённая логика `processingOnSession/NotOnSession`.
- Разбор текста: `split(" ")` — не учитывает множественные пробелы, табы,
  переводы строк; префикс `/` команды не вырезается.
- `data` для сообщения — `String[]`, а для callback — `String` (несогласованно).
- Логирование текста сообщений на INFO (PII).
- Много `if (sessionManagerService == null)` — следствие опционального бина.
**Перенос:** разнести на «парсер апдейта → контекст» и «диспетчер сессий».
Нормализовать разбор аргументов (regex по whitespace, отрезать `/` и `@bot`).
Согласовать тип аргументов. Понизить логи. Вынести null-логику сессий в
«no-op» реализацию (Null Object), чтобы убрать ветвления.

## 5. Сессии

### `SessionManagerService` (интерфейс)
**Хорошо:** есть абстракция → можно подменить реализацию.
**Перенос:** оставить интерфейс; добавить no-op реализацию по умолчанию.

### `SessionManagerServiceImpl`
**Плохо:**
- **Единицы времени:** конфиг и логи говорят «секунды», а планировщик —
  `TimeUnit.MINUTES`. Рассинхрон семантики.
- `timeLife/2`, `timeLife/4` для `initialDelay`/`period`: при малых значениях
  `period` может стать `0` → `IllegalArgumentException`.
- `synchronized (chatBotMap)` поверх `ConcurrentHashMap` — бессмысленно и
  частично вредно (сводит на нет конкурентность).
- `getAndCreate` делает `get`-затем-`put` под `synchronized`, тогда как есть
  `computeIfAbsent`.
- TTL считается в миллисекундах (`System.currentTimeMillis()`), а конфиг —
  «минуты/секунды»: легко ошибиться при сравнении (`create + life`).
**Перенос:** использовать `Duration`. Очистку делать корректно
(`computeIfAbsent`, без внешнего `synchronized`, без деления, дающего 0).
Согласовать единицы измерения по всей цепочке.

### `ChatBotSession`
**Плохо:** `getUser` — `get`-затем-`put` (гонка) несмотря на `ConcurrentHashMap`.
**Перенос:** `usersMap.computeIfAbsent(userId, k -> supplier.get())`.

### `UserBotSession`
**Плохо:** `data` синхронизировано, остальные поля — нет (несогласованная
потокобезопасность). `lastMessage` (`ArrayList`) не потокобезопасен.
**Перенос:** определиться с моделью потокобезопасности (например, доступ к
сессии одного пользователя сериализован) и применить её единообразно.

## 6. Callback

### `CallbackData` / `CallbackDataSender`
**Назначение:** `callback_data` хранится как JSON `{command, data}`.
**Плохо:**
- **Главное:** Telegram ограничивает `callback_data` 64 **байтами**. JSON с
  ключами `command`/`data` почти всегда вылезет за лимит → ошибка отправки.
- `data` — только `String`.
**Перенос:** отказаться от «толстого» JSON. Варианты: компактный формат
(`cmd:arg`), числовые/короткие коды команд, либо хранить полезную нагрузку на
сервере и класть в callback короткий ключ. Обязательно валидировать длину ≤ 64
байта при формировании кнопки.

## 7. Утилиты

### `KeyboardUtil`
**Хорошо:** удобные фабрики inline/reply клавиатур.
**Плохо:** `InlineKeyboardButton.builder().build()` затем сеттеры — нелогично
(можно сразу в билдере). Опять JSON-callback с риском > 64 байт. Нет проверок на null.
**Перенос:** оставить, но строить кнопку целиком через билдер, валидировать
callback по длине, добавить null-safe.

### `MediaGroupStorage`
**Назначение:** собирает альбомы (media group) по `mediaGroupId`.
**Хорошо:** аккуратная работа с `InputMedia`, выбор крупнейшего `PhotoSize`.
**Плохо:** не бин (создаётся вручную). Нет механизма «альбом собран» — Telegram
шлёт элементы альбома отдельными апдейтами с задержкой; здесь только накопление
без флаша/колбэка по готовности. `ArrayList` в `computeIfAbsent` — ок только
потому, что polling однопоточный.
**Перенос:** хорошая утилита; добавить дебаунс/таймер «альбом готов → колбэк»,
сделать опциональным бином.

### `util/CacheMap`
**Назначение:** `Map` с TTL и фоновой очисткой.
**Плохо:**
- Нарушает контракт `Map`: `keySet()` возвращает **копию** (snapshot), а не view;
  `entrySet().setValue` не пишет обратно в карту.
- `scheduleAtFixedRate(... time/5 ...)`: при `time < 5` период `0` → исключение.
- Каждый экземпляр планирует задачу на общем статическом экзекьюторе, но
  **никогда её не отменяет** → утечка при множественном создании.
- В целом переусложнён.
**Перенос:** заменить на готовое решение (Caffeine `expireAfter*`) либо на
существенно более простую обёртку. Если оставлять свой — чинить контракт `Map`,
деление на 0, отмену задач (`close()`/`Cleanable`).

### `util/OutConsoleCopyQueue`
**Назначение:** перехват `System.out` в очередь (для показа консоли где-то ещё).
**Плохо:** глобально подменяет `System.out` (статический сайд-эффект); сомнителен
в составе bot-библиотеки; небезопасный «синглтон» `instanceDefaultConsoleCopy`.
Тесты на него зависят от глобального состояния и порядка.
**Перенос:** **не переносить** в библиотеку. Это инфраструктура приложения, не
бота. Если нужно — отдельный модуль/проект.

## 9. Сборка

### `build.gradle.kts` (новый проект)
**Замечания:**
- Плагин `org.springframework.boot` по умолчанию собирает исполняемый
  (fat) jar — для **библиотеки** нужно отключить `bootJar` и включить обычный
  `jar` (или использовать только `io.spring.dependency-management` без
  `bootJar`-сборки).
- Jackson (`ObjectMapper`) используется в коде, но явной зависимости нет —
  `spring-boot-starter` его не тянет. Нужно добавить `jackson-databind`
  (или `spring-boot-starter-json`), иначе `NoClassDefFoundError`.
- Spring Boot `4.1.0` — очень свежий; убедиться в совместимости с
  `telegrambots` 10.0.0.
- SnakeYAML используется напрямую — приходит транзитивно от Spring Boot, но
  лучше объявить явно, если оставлять `TelegramBotConfig`.
