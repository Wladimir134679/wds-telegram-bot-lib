## Этап 0. Фундамент сборки
- [ ] Настроить сборку **библиотеки**: отключить `bootJar`, включить обычный `jar`.
- [ ] Добавить явную зависимость Jackson (`spring-boot-starter-json` или
      `jackson-databind`) — её использует callback/keyboard.
- [ ] Проверить совместимость Spring Boot 4.1.0 ↔ `telegrambots` 10.0.0.
- [ ] Завести пакет новой библиотеки (например `ru.wds.smp.telegram...`).

## Этап 1. Авто-конфигурация (правильный starter)
- [ ] Создать `@AutoConfiguration`-класс вместо `@ComponentScan`.
- [ ] Бины объявлять явно через `@Bean` + `@ConditionalOnMissingBean` /
      `@ConditionalOnProperty`, чтобы потребитель мог переопределить.
- [ ] `@ConfigurationProperties` подключить через `@EnableConfigurationProperties`
      в авто-конфиге (без сканирования пакетов потребителя).
- [ ] Прописать класс в `AutoConfiguration.imports`.
- [ ] Команды/обработчики собирать через инъекцию `List<HandlerBotUpdate>` /
      поиск бинов с `@CommandNames` в **контексте потребителя**, не своим scan'ом.

## Этап 2. Движок и жизненный цикл
- [ ] Вынести Telegram-клиент в отдельный бин (фабрика по токену) —
      тестируемость.
- [ ] Диспетчер апдейтов: не мутировать инжектированный список (копия + сорт),
      обернуть каждый обработчик в try/catch (изоляция ошибок).
- [ ] Жизненный цикл бота — `SmartLifecycle` (start/stop), убрать
      `@PostConstruct`+`@EventListener` дубль.
- [ ] Обработать неуспешную регистрацию и null-сессию (без NPE при stop).
- [ ] Логирование: не печатать весь `Update`/текст/токен; уровни DEBUG.

## Этап 3. Маршрутизация команд (ядро подхода)
- [ ] При регистрации команды **один раз** находить и кэшировать `Method`
      `@CommandFirst`/`@CommandOther` (+ преднасчитанные резолверы параметров).
- [ ] Детерминировать поведение при нескольких аннотированных методах
      (запрет/ошибка/документированный выбор).
- [ ] Падать с понятной ошибкой при конфликте имён команд.
- [ ] Расширить резолверы параметров: текстовые аргументы команды, числа,
      возможность кастомных конвертеров.
- [ ] Нормализовать разбор: whitespace-regex, отрезать `/` и `@botname`.
- [ ] Согласовать тип полезной нагрузки (`data`) между MESSAGE и CALLBACK.
- [ ] Прокидывать исключения пользовательского кода без потери причины.

## Топ-критичных багов, которые точно надо исправить

1. **Логирование токена бота** (`TelegramBotProperty.postInit` через `@ToString`).
2. **`callback_data` > 64 байт** (JSON-сериализация) — упадёт в проде.
3. **Единицы времени TTL** (секунды в конфиге vs `TimeUnit.MINUTES`) + возможный
   период `0` → `IllegalArgumentException` при старте.
4. **Двойной старт бота** (`@PostConstruct` + `@EventListener`).
5. **NPE при остановке**, если регистрация упала (`botSession == null`).
6. **Нет изоляции ошибок** в цепочке обработчиков (`forEach` без try/catch).
7. **Гонки в сессиях** (`get`-затем-`put`, `synchronized` поверх CHM).
8. **Рефлексия команд без кэша** и без гарантии порядка `getDeclaredMethods()`.

Начинать рекомендую с Этапа 0–1 (правильный starter), затем сразу закрыть
критичные баги №1–3, и только потом переносить ядро маршрутизации (Этап 3),
ради которого весь подход и затевался.
