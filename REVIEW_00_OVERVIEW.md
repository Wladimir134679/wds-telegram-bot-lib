# Разбор старого проекта `old_project` — обзор и архитектура

> Цель документа — критически оценить старый код перед переносом подхода в новую
> библиотеку `wds-telegram-bot-lib`. Здесь общая картина; детальный поклассовый
> разбор — в [REVIEW_01_CLASSES.md](REVIEW_01_CLASSES.md); план переноса и
> исправлений — в [REVIEW_02_MIGRATION.md](REVIEW_02_MIGRATION.md).

## Что это за проект

Это Spring Boot **starter** (библиотека авто-конфигурации) для написания
Telegram-ботов. Идея здравая и мне самому нравится: ты подключаешь зависимость,
а дальше пишешь «контроллеры команд» почти как в Spring MVC, с аннотациями и
инъекцией параметров по типу. Используется `telegrambots-longpolling` 10.0.0
(long polling, без webhook).

## Ключевые концепции (это ядро подхода, его и переносим)

| Концепция | Класс(ы) | Суть |
|---|---|---|
| Цепочка обработчиков | `HandlerBotUpdate`, `TelegramLongPollingEngine` | Каждый `Update` прогоняется через список обработчиков, отсортированных по `priority()` |
| Движок/клиент | `TelegramLongPollingEngine` | Реализует `LongPollingSingleThreadUpdateConsumer`, оборачивает `TelegramClient`, рассылает апдейты обработчикам |
| Жизненный цикл бота | `TelegramBotApiStarterService` | Регистрация/остановка бота, реакция на события start/stop |
| Маршрутизация команд | `@CommandNames`, `@CommandFirst`, `@CommandOther`, `CommandContainer`, `Manager*` | Бины с `@CommandNames` находятся через контекст, методы вызываются рефлексией с инъекцией аргументов по типу |
| Сессии | `SessionManagerService(Impl)`, `ChatBotSession`, `UserBotSession` | Состояние диалога по чату и пользователю с TTL-очисткой |
| Callback-данные | `CallbackData`, `CallbackDataSender` | `callback_data` сериализуется в JSON |
| Утилиты | `KeyboardUtil`, `MediaGroupStorage`, `CacheMap`, `OutConsoleCopyQueue` | Клавиатуры, сбор альбомов, кэш с TTL, перехват консоли |
| Конфигурация | `TelegramBotProperty`, `TelegramBotConfig`, `InitTelegramBotStarter` | Свойства бота + отдельный YAML-файл конфига |

## Что хорошего (это стоит сохранить)

1. **Сама идея starter'а** — подключил зависимость, описал команды, бот работает.
   Декларативность в стиле Spring MVC — сильная сторона.
2. **Цепочка обработчиков с приоритетами** — расширяемо: логирование, метрики,
   обработка команд, антифлуд — всё это просто новые `HandlerBotUpdate`.
3. **Инъекция аргументов команды по типу** (`Update`, `Message`, `CallbackQuery`,
   `Chat`, `UserBotSession`, …) — удобно, читабельные сигнатуры методов-команд.
4. **Опциональность фич через `@ConditionalOnProperty`** — сессии, конфиг-файл
   включаются флагом. Правильное направление мысли для библиотеки.
5. **Сессии с TTL и фоновой очисткой** — продуманная идея (чат → пользователи),
   `@Autowired(required = false)` делает сессии необязательными.
6. **Разделение `MESSAGE`/`CALLBACK`** на уровне типа команды.

## Главные архитектурные проблемы (по убыванию важности)

1. **Перепутаны роли «приложение» и «библиотека».** В библиотеке есть
   `main()`, `@SpringBootApplication`, `@ComponentScan`, `@EntityScan`. Стартер
   так делать не должен — он навязывает потребителю сканирование пакетов и тянет
   лишнее. Нужен современный `@AutoConfiguration` + `@ConditionalOn*`.
2. **`callback_data` сериализуется в JSON, но Telegram ограничивает его 64 байтами.**
   `{"command":"...","data":"..."}` легко вылезает за лимит → ошибка в рантайме.
   Это концептуальный изъян, а не мелочь.
3. **Диспетчеризация команд через рефлексию без кэширования.** На каждый вызов
   команды делается `getClass().getDeclaredMethods()` и линейный поиск аннотаций.
   Порядок методов не гарантирован JVM. Сканирование надо делать один раз при старте.
4. **Проблемы потокобезопасности в сессиях.** `synchronized` поверх
   `ConcurrentHashMap` (бессмысленно), `get`-затем-`put` вместо `computeIfAbsent`
   (гонка), несогласованная синхронизация полей.
5. **Семантический баг в TTL сессий.** Конфиг и логи говорят про секунды, а
   планировщик использует `TimeUnit.MINUTES`. При малых значениях `timeLife/4`
   может стать `0` → `IllegalArgumentException` в `scheduleAtFixedRate`.
6. **Нет изоляции ошибок в цепочке обработчиков.** `forEach` без try/catch:
   исключение в одном обработчике рушит обработку всего апдейта.
7. **Шумное и небезопасное логирование на уровне INFO.** Логируется весь `Update`,
   текст сообщений, а в `TelegramBotProperty` через `@ToString` — ещё и **токен бота**.
8. **Запуск бота в `@PostConstruct` + дублирование с `@EventListener`.** Один и
   тот же `startBot()` вызывается двумя путями → возможен двойной старт.
   Жизненным циклом должен заниматься `SmartLifecycle`/`ApplicationRunner`.

## Итоговая оценка по слоям

| Слой | Оценка | Комментарий |
|---|---|---|
| Идея/архитектура в целом | 👍 хорошо | Концепция starter'а + контроллеры команд правильная |
| Реализация авто-конфигурации | 👎 плохо | Старый стиль (`@ComponentScan`), смешение с приложением |
| Жизненный цикл бота | ⚠️ средне | Работает, но хрупко (PostConstruct, двойной старт, глотание ошибок) |
| Маршрутизация команд | ⚠️ средне | Удобно снаружи, но рефлексия без кэша и без гарантий порядка |
| Сессии | ⚠️ средне | Хорошая идея, но баги потокобезопасности и единиц времени |
| Callback | 👎 плохо | Игнорирует лимит 64 байта |
| Утилиты | ⚠️ разное | `KeyboardUtil`/`MediaGroupStorage` ок, `OutConsoleCopyQueue`/`TelegramBotConfig` сомнительны |
| Тесты | 👎 плохо | Есть только на `OutConsoleCopyQueue`, и те зависят от глобального состояния |

## Сквозные замечания (касаются всего кода)

- Опечатка в имени класса `WdsTelegramBotApplicaton` (пропущена `i`).
- Именование: `executeNotException` на самом деле **бросает** исключение;
  обработчики называются `...ServiceImpl`, хотя это не сервисы.
- Местами `System.out.println` вместо логгера (`TelegramBotConfig`).
- Lombok-аннотации навешаны избыточно и противоречиво (`@Data` + `@Getter` +
  `@Setter` вместе; `@Data`/`@ToString` на пустых классах-событиях).
- Jackson (`ObjectMapper`) используется, но явной зависимости в новом
  `build.gradle.kts` нет — нужно проверить classpath.
- Плагин `org.springframework.boot` по умолчанию собирает исполняемый jar —
  для библиотеки это нужно настроить (`bootJar`/`jar`).
